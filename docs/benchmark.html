Analysis

Throughput

Fan-out 499에서 15,000 pps, 145 Mbps 처리. GbE 대역폭(1 Gbps) 대비 14.5% 사용으로 NIC 병목 아님. rx_total 패킷 수가 tx × fan-out과 정확히 일치 (오차 < 0.01%).

Packet Loss

전체 13회 테스트 중 loss 발생 3회. 모두 subscriber당 1패킷 단발성 — burst loss 없음. Fan-out 499에서도 15/898,576 = 0.002%. 서버 CPU 포화가 아닌 네트워크 계층 일시적 경합.

Subscriber Fairness

Fan-out 루프가 순차적으로 499회 encrypt + send_to를 수행하므로 앞순번(~5ms)과 뒷순번(~25ms) 사이 ~20ms 편차 발생. 실시간 통화 기준(150ms) 대비 전 subscriber가 충분히 낮음.

Memory

499 세션에 45MB RSS — 세션당 ~90KB. GC 없는 Rust 특성. SRTP context + RTP cache + WS 버퍼 포함. RPi 4GB RAM 기준 이론적 최대 ~40,000 세션 (메모리만 고려 시).

Why This Fast

Rust + Tokio — GC 없음, async/await 비동기 I/O, zero-cost abstraction
Single UDP Socket — 소켓 관리 오버헤드 없음, 커널 버퍼 1개만 사용
O(1) Lookup — DashMap 3중 인덱스 (user_id, ufrag, addr) fan-out 루프 내 조회 비용 최소화
SRTP Only Cost — hot-path에서 AES-128-CM encrypt만 실행. 나머지는 async spawn
Ring Buffer Cache — NACK/RTX 재전송용 RTP 캐시가 fan-out에 추가 메모리 부담 없음

Fan-out Limitations

본 테스트는 1 pub → N sub (단일 스트림 fan-out)만 측정. 실제 N인 화상회의에서는 N pub × (N-1) sub = N² 스트림으로 서버 부하 구조가 근본적으로 다릅니다. 아래 Conference 벤치마크에서 실제 회의실 시나리오를 측정했습니다.

Conference Analysis

N² Scaling

5→25인: CPU가 스트림 수에 비례하여 선형 증가. 20인(380 스트림)에서 CPU 52% — 설계 목표 규모에서 충분한 여유. 25인(600 스트림)에서 CPU 80%로 한계 접근.

30인 Cliff

30인에서 loss 9.6%, avg 67ms로 급락. 원인: UDP hot loop가 단일 async task → Cpu0 한 코어에 91.6% 집중. 나머지 3코어는 유휴. fan-out 루프 병렬화로 개선 가능.

Fairness

모든 PASS 라운드에서 참가자 간 latency 편차 < 1.2ms. N이 증가해도 공평성 유지 — fan-out 루프 순서 편차가 참가자 수에 의해 평균화.

Recommended

20인 이하: 라즈베리파이 권장 (CPU 52% 여유).
25인: 한계 접근, 모니터링 필요.
30인+: x86 서버 또는 코어 분산 최적화 필요.

Next Steps

Fan-out 루프 병렬화 (멀티코어 활용 → 30인+ 가능성), TWCC 구현 후 전후 비교, 원격 네트워크(LTE/공인 IP) 테스트, x86 서버 벤치마크 비교.

Loss Reduction

30인 conference에서 각 단계별 loss 감소 추이. W-3에서 0%로 수렴.

Optimization Analysis

W-1: Fan-out Spawn

encrypt+send를 tokio::spawn으로 분리. 메인 recv 루프가 fan-out에 블로킹되지 않아 Tokio work-stealing이 유휴 코어를 활용. loss 9.6% → 1.3%.

W-2: SO_REUSEPORT

동일 포트에 4개 독립 소켓, 커널 4-tuple hash로 패킷 분배. recv+decrypt도 멀티코어 분산. loss 1.3% → 0.1%.

W-3: Egress Task

subscriber별 전용 egress task, outbound_srtp 독점. Mutex 경합 원천 제거. LiveKit 패턴. loss 0.1% → 0.000%, latency 15ms.
