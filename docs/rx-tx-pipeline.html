Rx Pipeline — 수신 경로

PTT 화자 전환 시 수신 오디오가 단조 증가하며 지연되는 문제를 해결한 과정. 18시간 디버깅 세션에서 3가지 접근법을 시도하여 최종 해결에 도달했습니다.

증상 — NetEQ Jitter Buffer 폭주

PTT 모드에서 화자 전환을 반복하면 수신 오디오 지연이 단조 증가. 30초 만에 1초 이상 누적되어 체감상 "말한 뒤 한참 뒤에 들림". 한 번 누적된 지연은 절대 줄어들지 않음 (shrink=0).

결론: 서버 clock과 클라이언트 벽시계는 독립이므로, 클라이언트 측 silence injection은 구조적으로 해결 불가.

✅ RESOLVED
최종 해결 — 서버 Dynamic ts_guard_gap

핵심 아이디어 — "ts를 확 늘리면 안되?"

NetEQ는 arrival_time gap과 ts gap의 불일치를 jitter로 해석합니다.

고정 ts_gap=960 (20ms)
expected = 960/48000 = 20ms
actual (idle 3초) = 3000ms
jitter = 3000 - 20 = 2980ms
→ buffer를 3초로 확장!
dynamic ts_gap (경과시간 × 48kHz)
ts_gap = 3000 × 48 = 144000
expected = 144000/48000 = 3000ms
actual = 3000ms
jitter = 3000 - 3000 = 0ms
→ buffer 안 키움!
구현 서버 ptt_rewriter.rs — clear_speaker()에서 cleared_at 기록, switch_speaker()에서 경과 시간 × 48kHz로 dynamic gap 계산. silence flush의 last_virtual_ts를 기준으로 추가 gap 적용. Audio만 (Video는 키프레임 게이팅이 처리).

교훈

수신자(NetEQ) 관점에서 역으로 사고하라

"패킷으로 빈 구간을 채워야 한다"는 프레임에 갇히지 말 것. NetEQ가 보는 것은 arrival_time과 ts gap의 일치 여부.

ts gap ≠ RFC 위반

ts gap이 큰 것은 "오래 침묵했다"를 의미하는 완전히 합법적인 RTP. RFC 3550에서 ts는 샘플 카운터이지 벽시계가 아니다.

shrink=0이 핵심 단서

"buffer를 줄일 이유가 없다"가 아니라 "줄일 수 없다" — arrival_time과 ts가 불일치하면 항상 "jitter가 크다"고 판단.

이중 clock 보정은 구조적 실패

서버 encoder clock과 클라이언트 벽시계는 독립. 매 전환마다 offset이 달라져 drift가 누적.

Video Rx — 비디오는 패킷 레벨 조작을 배제합니다. 키프레임 확보는 아래 Client-driven Readiness 프로토콜에서 처리합니다.

연결 유지 신호 주기를 극단적으로 늘려 모뎀 웨이크업을 최소화합니다. 미디어 엔진 자체는 내리지 않습니다 — 암호화 키와 인증서가 메모리에서 소실되면 전체 보안 핸드셰이크를 재수행해야 하기 때문입니다.

NAT 매핑 소실 가능 — 장주기 유지 시 네트워크 경로가 만료될 수 있습니다. 이 경우 연결 재협상이 필요하며, 10분+ 미사용자에게 1~2초 복구는 허용 가능한 수준입니다.

NAT Traversal Optimization

미디어 미전송 상태에서도 연결 유지 신호가 주기적으로 모뎀을 깨웁니다. FSM 단계별로 신호 주기를 동적 조정하여 배터리 소모를 최소화합니다.

실측 필수 — 국내 주요 통신사(SKT / KT / LGU+)의 LTE / 5G 환경별 NAT 타임아웃은 20~60초로 편차가 큽니다. 연결 유지 주기는 실측 데이터 기반으로 튜닝합니다.

기존 방식의 문제

서버가 타이밍을 추측하여 키프레임 요청을 보냅니다. 카메라 AE/AWB 수렴 전에 요청이 도착하면 허공에 낭비되고, 수렴 후 한참 뒤에 도착하면 불필요한 지연이 발생합니다.

Client-driven 방식의 이점

클라이언트가 "첫 프레임 캡처 완료" 콜백을 받은 시점에 서버에 준비 완료를 알립니다. 서버의 타이밍 추측이 불필요하며, 첫 프레임 캡처 = AE 1차 수렴이므로 키프레임 요청 2발이면 충분합니다.

Exception Handling — Signaling Disconnection

시그널링 채널(TCP)이 단절되면, 모든 캐시된 상태를 즉시 파기하고 완전한 재시작을 수행합니다.

Zero Tolerance — Full Cold Start

시그널링 연결이 유실되면 SDP, 암호화 컨텍스트, 연결 상태 등 모든 캐시를 즉시 파기합니다.

이후 이벤트 발생 시 완전히 새로운 세션을 수립합니다 — 서버-클라이언트 상태 불일치(좀비 세션)를 원천 차단합니다.

타협 없음.

Technology Foundation

본 파이프라인은 서버 측 RTP 리라이팅과 클라이언트 표준 WebRTC API의 조합으로 구현됩니다. 수신 오디오 지연 문제는 libwebrtc C++ 패치(AudioInterceptor)를 시도했으나 이중 clock drift 문제로 폐기하고, 서버 ptt_rewriter의 dynamic ts_gap 조정으로 최종 해결했습니다. AudioInterceptor 패치는 libwebrtc 커스텀 빌드에 잔존하나 비활성화 상태입니다.
