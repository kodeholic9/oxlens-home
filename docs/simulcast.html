Simulcast —
하나의 카메라, 두 개의 품질

Simulcast는 하나의 카메라로 고화질(h)과 저화질(l) 두 레이어를 동시에 인코딩하여 서버에 전송하는 기술이다. SFU는 수신자의 네트워크 상태, 화면 크기, 중요도에 따라 적절한 레이어를 선택하여 전달한다. 0xLENS는 Phase 0부터 Phase 3까지 4단계에 걸쳐 Simulcast를 구현했다.

2026-03-20 · 설계 + 구현 + 실측

00 Simulcast란 무엇인가

Without Simulcast

카메라 → 단일 스트림(예: 1080p 2Mbps) → SFU → 모든 수신자에게 동일 품질 전달.

WiFi 좋은 수신자와 3G 수신자가 같은 스트림을 받는다. 3G 수신자는 패킷 손실과 버퍼링에 시달리고, SFU가 대역폭 제한을 걸면 WiFi 수신자도 저화질을 감수해야 한다.

Camera → [1080p 2Mbps] → SFU → [1080p] → 🖥️ WiFi ✅
→ [1080p] → 📱 3G ❌

With Simulcast

카메라 → 고화질(h) + 저화질(l) 두 레이어 동시 인코딩 → SFU가 수신자별로 최적 레이어 선택.

WiFi 수신자는 h 레이어(1080p)를, 3G 수신자는 l 레이어(360p)를 받는다. 각자의 네트워크에 맞는 최적 품질.

Camera → [h 1080p 1.6M] → SFU → [h] → 🖥️ WiFi ✅
[l 360p 180k] ↗→ [l] → 📱 3G ✅

핵심 개념

h (High)

고화질 레이어. 원본 해상도 + 높은 비트레이트. 발표자, 메인 화면, WiFi 환경에 적합. 0xLENS 기준 1.6Mbps / 24fps.

l (Low)

저화질 레이어. 축소 해상도 + 낮은 비트레이트. 썸네일, 작은 타일, 모바일 환경에 적합. 0xLENS 기준 180kbps / 24fps.

pause

영상 수신 중단. 화면에 보이지 않는 참가자의 대역폭을 절약. IntersectionObserver 등으로 자동 제어 가능.

01 SFU에서 Simulcast가 어려운 이유

SSRC 문제 — Chrome이 2개를 보내지만 Subscriber는 1개만 알아야 한다

Publisher Chrome은 h 레이어와 l 레이어에 각각 다른 SSRC를 할당한다. 그런데 Subscriber Chrome의 SDP에는 하나의 video m-line만 있고, 하나의 SSRC만 매칭할 수 있다. SFU가 레이어를 전환할 때마다 SSRC가 바뀌면 Chrome은 "새로운 스트림"으로 인식하여 디코더를 재생성한다. 이는 화면 깜빡임, 디코딩 지연, 심하면 영상 멈춤을 유발한다.

Sequence / Timestamp 연속성 — 끊김 없는 전환

h → l 전환 시, l 레이어의 seq/ts는 h와 완전히 다른 시퀀스를 갖는다. 그대로 전달하면 Chrome의 jitter buffer가 "패킷 대량 손실"로 오판하여 NACK 폭풍 + FEC 요청을 쏟아낸다. SFU가 seq/ts를 연속적으로 rewrite해야 전환이 투명하게 느껴진다.

키프레임 동기화 — 레이어 전환은 키프레임에서만 가능

VP8/H.264의 P-frame은 이전 프레임과의 차분만 담고 있어, 새 레이어의 P-frame을 갑자기 보내면 디코더가 깨진다. 반드시 키프레임(I-frame)이 도착한 후에야 새 레이어 relay를 시작할 수 있다. PLI(Picture Loss Indication)로 키프레임을 요청하고, 도착할 때까지 P-frame은 전부 드롭해야 한다.

02 0xLENS의 해법 — 가상 SSRC + SimulcastRewriter

가상 Video SSRC 아키텍처

Publisher[h] real_ssrc=0xAAA ─┐
Publisher[l] real_ssrc=0xBBB ─┤  서버: ensure_simulcast_video_ssrc() → virtual=0xCCC
                               │
TRACKS_UPDATE(add) ────────────┤→ subscriber에게 ssrc=0xCCC (rid 제거)
ingress fan-out ───────────────┤→ SimulcastRewriter.rewrite(real→virtual)
subscriber Chrome ─────────────┘  SDP에 0xCCC → 매칭 ✓

가상 SSRC (Virtual SSRC)

Publisher별 고정 가상 video SSRC를 할당한다. TRACKS_UPDATE, ROOM_JOIN, ROOM_SYNC, TRACKS_ACK — 모든 시그널링 경로에서 실제 SSRC 대신 가상 SSRC를 전달한다. Subscriber Chrome은 항상 하나의 SSRC만 보므로 SDP 매칭이 보장된다.

SimulcastRewriter

Subscriber별 × Publisher별로 독립된 rewriter 인스턴스. RTP 패킷의 SSRC를 가상 값으로, seq/ts를 연속 offset으로 in-place rewrite한다. 레이어 전환 시 키프레임 도착까지 P-frame을 드롭하고, 도착 시점에 새 offset을 계산한다.

PLI 교착 방지 (2중 대책)

대책 #1: ingress fan-out에서 SubscribeLayer 즉석 생성 시 즉시 PLI burst [0, 200, 500, 1500]ms. 대책 #2: SUBSCRIBE_LAYER에서 old==new && !initialized → PLI burst. v0.6.3에서 자가 치유 추가: rewriter 미초기화 상태가 2초 이상 지속되면 자동 PLI 재시도.

PLI/NACK 역매핑

Subscriber가 PLI를 보내면 가상 SSRC → 현재 레이어의 실제 SSRC로 변환 후 Publisher에게 전달. NACK도 가상 seq → 실제 seq로 역산하여 올바른 RTX 패킷을 캐시에서 찾는다.

03 구현 Phase — 0에서 3까지

DONE Phase 0 — 구조 준비

simulcast_enabled 하드 플래그 도입. OFF 방은 기존 코드 100% 동작, ON 방만 새 경로 진입. 기존 Conference/PTT E2E 테스트 통과 필수 — 절대 기존 기능을 깨뜨리지 않는 원칙.

DONE Phase 1 — 멀티 레이어 수신 + 하이 레이어 전용 릴레이

서버: rid("h"/"l")로 트랙 구분, h 레이어만 fan-out. l 레이어 수신하되 릴레이하지 않음. TRACKS_UPDATE에서 rid="l" 필터링.

클라이언트: server-offer → client-offer 전환. Chrome이 SimulcastEncoderAdapter를 활성화하려면 addTransceiver에서 sendEncodings를 지정해야 하고, 이는 client-offer에서만 가능.

TWCC 수정: client-offer 전환 후 Chrome이 TWCC extmap ID=3을 할당, 서버 하드코딩 ID=6과 불일치 → 동적 참조로 수정.

RESERVED Phase 2 — IntersectionObserver 기반 자동 레이어 선택

화면에 보이는 타일만 h, 스크롤 밖 타일은 l 또는 pause로 자동 전환. Phase 3 기반 위에 클라이언트 로직만 추가하면 되므로 서버 변경 없음. 후순위.

DONE Phase 3 — 가상 SSRC + SimulcastRewriter + 동적 레이어 전환

서버 6파일 수정: participant.rs (SimulcastRewriter, SubscribeLayerEntry, ensure_simulcast_video_ssrc), room.rs (find_publisher_by_vssrc), message.rs, track_ops.rs (handle_subscribe_layer), helpers.rs, ingress.rs (simulcast fan-out + PLI 역매핑).

SSRC 참조 경로 10항목: PUBLISH_TRACKS, ROOM_JOIN, ROOM_SYNC, ROOM_LEAVE, SUBSCRIBE_LAYER, ingress fan-out, PLI relay, NACK 역매핑, TRACKS_ACK, SR relay — 전부 가상 SSRC로 교체.

클라이언트: SUBSCRIBE_LAYER(op=51) 시그널링 + long-press 타일 팝업 (h/l/pause 선택).

TODO Phase 4 — Active Speaker 자동 전환

RFC 6464 audio level 헤더 기반으로 현재 발화자를 서버가 감지. ACTIVE_SPEAKERS(op=144)를 200~500ms 주기로 브로드캐스트. 클라이언트가 발화자는 h, 나머지는 l로 자동 전환.

04 발견된 버그 & 해결 — 실전 검증 기록

BUG SUBSCRIBE_LAYER ↔ PUBLISH_TRACKS 레이스 컨디션

ROOM_JOIN 후 기존 참가자가 SUBSCRIBE_LAYER를 보내는 시점에 새 참가자의 PUBLISH_TRACKS가 아직 미도착. subscribe_layers entry가 먼저 생성 → ingress에서 created=false → PLI 안 보냄 → 영상 영원히 안 뜸.

FIX: PLI 자가 치유 — rewriter 미초기화 + 2초 경과 시 자동 PLI 재시도 (needs_pli_retry)
BUG 퇴장 후 재입장 시 video fan-out 실패

퇴장 시 다른 참가자의 subscribe_layers에 이전 세션 entry 잔존. 재입장 시 old rewriter(initialized=true + 구 vSSRC) → PLI 안 보냄 + SSRC 불일치 → 영상 안 뜸.

FIX: purge_subscribe_layers() — ROOM_JOIN / ROOM_LEAVE / cleanup 3곳에서 stale entry 제거 (멱등)
NOISE PLI 유령 카운팅 (pli_sent=44/3s)

pli_sent 글로벌 카운터가 publisher 매칭 전에 증가 → publisher를 못 찾아 드롭된 PLI도 카운트 → 어드민에 유령 44/3s.

FIX: pli_sent를 publisher 매칭 성공 + 전송 성공 시에만 증가. 드롭 시 warn 로그.
