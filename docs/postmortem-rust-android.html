POST-MORTEM 2026-03-14

Rust + Android + libwebrtc

크로스 바운더리 삽질기 — 왜 실패했고, 무엇을 배웠고, 어디로 가는가

01. 무엇을 하려 했는가

목표: Rust 코어 하나로 Android / iOS / Desktop 모든 플랫폼에서 동일한 PTT(Push-to-Talk) SDK를 제공한다.

// 구상했던 아키텍처
oxlens-sdk-core (Cargo workspace)
├── oxlens-core        — PTT 로직, 시그널링, SDP 빌더
├── oxlens-webrtc     — libwebrtc Safe Rust 래퍼
├── oxlens-webrtc-sys — C++ FFI (livekit webrtc-sys)
└── oxlens-jni        — Android JNI 브릿지 (cdylib)

libwebrtc C++ 정적 라이브러리(.a)를 Rust cdylib(.so) 안에 static link하여, 단일 liboxlens_jni.so (20MB)를 Android에 배포하는 구조.

02. 타임라인

3/12 성공

JNI 스캐폴딩 + Kotlin 래퍼 설계

oxlens-jni crate, OxLensClient.kt, OxLensEventListener.kt 완성

3/14 오전 성공

Android 크로스 빌드 성공

OpenSSL→rustls 전환, webrtc-sys 0.2/0.3 충돌 해결, libunwind 우회, NDK sysroot 패치. liboxlens_jni.so 20MB 생성.

3/14 오후 SIGSEGV

.so 로딩 즉시 크래시 — getauxval null deref

signal 11 (SIGSEGV), fault addr 0x0 — System.loadLibrary() 호출 시 .so constructor 단계에서 터짐

#00 getauxval+28                ← compiler_builtins의 poisoned stub
#01 __cxa_atexit+596           ← atomics 내부 함수
#02 _GLOBAL__sub_I_webrtc.cpp+80 ← libwebrtc C++ 글로벌 생성자
#03 call_constructors          ← 링커가 .so 생성자 호출 중
시도 A 실패

-Ctarget-feature=-outline-atomics

pre-compiled std에 이미 getauxval 심볼이 포함되어 있어 cdylib에 그대로 끌려 들어옴. 우리 코드에만 적용되므로 무의미.

시도 B 실패

__getauxval() C 함수 직접 제공

compiler_builtins의 getauxval이 내부적으로 호출하는 weak extern __getauxval을 C로 구현. 링커 심볼 resolve 순서 문제로 우리 함수가 무시됨.

시도 C 회피 가능

Rust 1.88 다운그레이드

PR #144938이 stable에 들어온 건 1.93(2026-03). 1.92 이하에서는 Android outline-atomics 비활성. 문제 자체가 발생하지 않음. 그러나 근본 해결이 아닌 회피.

3/14 결정 PIVOT

Kotlin + libwebrtc Java API 전환

Rust FFI 경로 포기. libwebrtc AAR(BSD 3-Clause)의 org.webrtc.* Java 바인딩을 Kotlin에서 직접 사용. 심볼 충돌 원천 차단.

03. 근본 원인 분석

세 생태계의 교차점

Rust 컴파일러, Android NDK, libwebrtc — 각각은 잘 작동하지만, 셋이 만나는 지점에서 아무도 테스트하지 않은 경로가 생긴다.

Rust 1.93

PR #144938으로 aarch64-linux-android에 outline-atomics 기본 활성화. compiler_builtins가 getauxval 심볼을 .so에 export.

Android Bionic

libc에 getauxval은 있지만, compiler_builtins가 참조하는 __getauxval(glibc ABI)은 없음. .so 내부 stub이 libc를 가림.

libwebrtc

C++ 글로벌 생성자(_GLOBAL__sub_I_webrtc.cpp)가 .so 로딩 시 getauxval을 호출. broken stub으로 진입 → null deref.

왜 아무도 안 만났는가

  • ▸ PR #144938은 2025-12 머지, Rust stable 1.93은 2026-03 릴리스 — 이번 달.
  • ▸ LiveKit Android SDK는 순수 Kotlin + Java 바인딩. Rust cdylib를 Android에서 안 씀.
  • ▸ LiveKit의 Rust SDK는 데스크톱/서버 전용. Flutter FFI도 아직 1.93으로 빌드 안 함.
  • ▸ 즉, Rust 1.93 + libwebrtc static link + Android cdylib 조합을 시도한 팀이 세상에 거의 없음.

.so 크기20MB

선례없음

Kotlin + AAR 경로의 이점

빌드 시간1~2분
AAR 크기OS 관리
디버깅Android Studio 네이티브
크로스 바운더리Kotlin ↔ Java (동일 런타임)
심볼 충돌원천 차단
선례LiveKit, Onesignal 등 다수
라이선스BSD 3-Clause (상용 자유)
하드웨어 코덱자동 지원

잃는 것 — 정말 잃는가?

▸ 멀티플랫폼 코어? — Android 우선. iOS는 Phase 3. 데스크톱은 Tauri WebView. 실제 필요 시점에 재검토.
▸ Rx/Tx Pipeline 설계? — 8개 항목 중 6개 그대로 적용. Tx FSM, 카메라 웜업은 Java API로 100% 가능. Rx offset은 서버에서 처리.
▸ 배터리 전략? — 핵심은 FSM 설계 자체이지 언어 선택이 아님. Java API의 RtpSender.encoding.active로 동일 구현.

05. 교훈

① 선례를 먼저 찾아라

크로스 바운더리 조합을 시도하기 전에 "이미 성공한 팀이 있는가"를 확인한다. 없으면 "왜 안 했는가"를 파악한다. LiveKit이 Rust + Android cdylib를 안 쓰는 데는 이유가 있었다.

② 설계의 아름다움에 끌려 현실을 무시하지 마라

"Rust 코어 하나로 전 플랫폼"은 아름다운 비전이지만, 당장 필요한 건 Android에서 돌아가는 PTT SDK다. Show me the money.

③ 크로스 바운더리 비용은 기하급수적이다

Rust 안에서, 또는 Kotlin 안에서는 각각 잘 돌아간다. 경계를 넘는 순간(Rust → C++ FFI → Android Linker) 디버깅 비용이 폭발한다. 경계를 줄이는 방향이 항상 옳다.

④ 삽질은 자산이다

이 과정에서 확인한 것: libwebrtc AAR + Java API가 안전한 경로, Rx/Tx Pipeline 설계가 Kotlin에서도 유효, 배터리 전략은 언어 무관. "왜 안 되는지"를 정확히 이해하고 넘어간 건 향후 의사결정의 자산.

⑤ Rust 생태계의 한계를 인지하라

서버(Rust + Tokio + Axum)와 웹(JS/Vue)은 성숙한 생태계에서 일했다. 모바일 + Rust + C++ FFI는 생태계의 가장자리. 문서도 없고 선례도 없고 커뮤니티 지원도 없다. 언어의 강점과 생태계의 성숙도는 별개 문제다.
