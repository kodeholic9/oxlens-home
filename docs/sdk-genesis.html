00 — PROLOGUE

출발점: Rust 코어 하나로 전 플랫폼을

서버는 이미 완성되어 있었다. Rust + Tokio + Axum으로 만든 SFU 서버는 라즈베리파이 한 대에서 30인 Conference를 loss 0.000%로 처리하고 있었다.

다음 목표는 명확했다 — Android에서 돌아가는 PTT SDK. Show me the money.

처음 구상한 아키텍처는 아름다웠다. Rust 코어 하나로 시그널링, SDP 빌더, Floor Control을 구현하고, 플랫폼별 얇은 JNI/FFI 레이어만 씌우면 Android, iOS, Desktop 전부 커버할 수 있다. libwebrtc C++ 정적 라이브러리를 Rust cdylib 안에 static link하면 단일 liboxlens_jni.so (20MB) 하나로 배포 완료.

"Rust 코어 하나로 전 플랫폼" — 아름다운 비전이었다. 문제는, 아름다운 비전과 돌아가는 소프트웨어는 다른 것이라는 점이었다.

02 — THE PIVOT

전환의 순간

⚡

3월 14일, 오후

signal 11 (SIGSEGV), fault addr 0x0 — System.loadLibrary() 한 줄에서 .so가 터졌다. libwebrtc C++ 글로벌 생성자가 Rust compiler_builtins의 깨진 getauxval stub을 호출하는 순간이었다.

시도 A — 컴파일러 플래그

-Ctarget-feature=-outline-atomics
→ pre-compiled std에 이미 심볼 포함. 무의미.

시도 B — C 함수 직접 제공

__getauxval() 구현 제공
→ 링커 심볼 resolve 순서 문제로 무효.

시도 C — Rust 다운그레이드

1.88로 내리면 문제 자체가 사라짐.
→ 근본 해결이 아닌 회피.

결정의 근거

▸ LiveKit Android SDK — 순수 Kotlin + Java API. Rust cdylib를 Android에서 안 쓴다.
▸ Rust 1.93 + libwebrtc + Android cdylib 조합을 시도한 팀이 세상에 거의 없다.
▸ Rx/Tx Pipeline 설계 8개 항목 중 6개가 Kotlin에서 그대로 적용 가능.
▸ 배터리 전략의 핵심은 FSM 설계 자체이지, 구현 언어가 아니다.

설계의 아름다움에 끌려 현실을 무시하지 마라. 당장 필요한 건 Android에서 돌아가는 PTT SDK다. Show me the money.

03 — AFTER THE PIVOT

전환 후 48시간 — 폭발적 진도

Kotlin + libwebrtc Java API로 전환한 뒤, 막혔던 모든 것이 풀렸다. 이틀 만에 Rust 4일치 기능을 넘어섰다.

E2E 동작 확인 — 최종 상태

✓ Android: CAMERA+AUDIO granted → connect → ROOM_CREATE → ROOM_JOIN
✓ server codecs: [opus, VP8] → 자동 카메라 초기화
✓ publish PC: 3 tracks (audio + video + RTX) → ICE CONNECTED
✓ subscribe PC: remote tracks 수신 → ICE CONNECTED
✓ 양방향 음성: Android ↔ 브라우저 청취 확인
✓ 양방향 영상: Camera2 → SFU → 브라우저 영상 수신
✓ PTT: TALK 길게 누름 → FLOOR_REQUEST → Granted → 발화 → Release
✓ Mute 3-state: Conference soft/hard + PTT audio 차단
✓ AudioSwitch: Speakerphone 우선, BT 지원
✓ TRACKS_ACK/RESYNC: subscribe 트랙 누락 복구 프로토콜

04 — CRASH STORIES
