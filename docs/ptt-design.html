3GPP TS 24.380 · MCPTT Floor Control · Virtual SSRC · Adaptive Timestamp Alignment

N명의 화자를
하나의 스트림으로.

무전기(PTT)에서 가장 어려운 문제는 화자가 바뀔 때마다 수신 측의 오디오가 끊기지 않게 하는 것입니다. 0xLENS는 서버에서 가상 SSRC 스트림을 생성하여, 수신 측 디코더가 화자 교체를 전혀 인지하지 못하게 합니다. 그리고 NetEQ의 jitter buffer에 정확한 타이밍 정보를 제공하여, 침묵 구간 후에도 지연이 누적되지 않게 합니다.

01 — CONFERENCE vs PTT

같은 서버, 두 가지 모드

Conference

전원 동시 발화. N명이 각자의 SSRC로 오디오/비디오를 전송합니다.

서버는 단순 fan-out: A의 패킷을 B, C, D에게 그대로 릴레이.

수신 측은 참가자별 별도 디코더로 동시 재생.

A(ssrc=100) ──→ SFU ──→ B 디코더(ssrc=100)
B(ssrc=200) ──→ SFU ──→ A 디코더(ssrc=200)
C(ssrc=300) ──→ SFU ──→ A 디코더(ssrc=300), B 디코더(ssrc=300)
PTT (Push-to-Talk)

1인 발화, N-1명 청취. 발화권(Floor)을 획득한 1명만 전송합니다.

서버는 가상 SSRC로 리라이팅하여 단일 스트림으로 통합합니다.

수신 측은 디코더 1개로 모든 화자의 음성을 재생.

A(ssrc=100) ──→ SFU rewrite ──→ 가상(ssrc=999) ──→ B,C
B(ssrc=200) ──→ SFU rewrite ──→ 가상(ssrc=999) ──→ A,C
// 수신 측은 항상 ssrc=999 하나만 봄

"Conference는 N개의 파이프를 그대로 연결한다. PTT는 N개의 파이프를 하나의 파이프로 합친다 — 이음새가 보이면 안 된다."

02 — FLOOR CONTROL

발화권 중재 — 3GPP TS 24.380 기반

무전기의 핵심은 "누가 지금 말할 수 있는가"를 중재하는 것입니다. 0xLENS의 Floor Control은 3GPP MCPTT(Mission Critical PTT) 표준의 용어와 상태 머신을 따릅니다.

GRANTED 상태에서만 마이크 unmute. Floor Ping 2초 주기로 서버에 생존 신호를 보냅니다. 서버가 5초간 Ping을 못 받으면 강제 Revoke합니다.

웹과 Android가 동일한 FSM. JavaScript(app.js)와 Kotlin(FloorFsm.kt) 양쪽 모두 같은 상태 전이 로직을 구현합니다.

왜 Floor Control은 WS(TCP) 경로인가

MBCP(Media Burst Control Protocol) over UDP도 구현되어 있습니다 — RTCP APP(PT=204) 패킷을 서버가 파싱합니다. 하지만 실전에서는 WS 시그널링 경로를 주력으로 사용합니다. 이유는 세 가지입니다:

① API 제약

libwebrtc PeerConnection에서 임의 RTCP APP 패킷을 주입하는 공개 API가 없습니다. 별도 UDP 소켓을 열면 NAT 홀펀칭 유지 부담이 생깁니다.

② 충분한 속도

발화권 결정은 10ms 이내면 충분합니다. WS 왕복(TCP)이 50ms여도 체감 차이가 없습니다. 음성 첫 패킷 도착이 병목이지, 시그널링이 병목이 아닙니다.

③ 서버 복잡도

UDP MBCP를 주력으로 쓰면 서버 demux에 RTCP APP 분리 로직이 hot path에 추가됩니다. WS는 시그널링 핸들러에서 처리하므로 미디어 경로와 완전 분리됩니다.

03 — VIRTUAL SSRC REWRITING

N명의 화자를 하나의 스트림으로 합치는 기술

PTT의 핵심 기술은 ptt_rewriter입니다. 여러 화자의 RTP 패킷을 하나의 가상 SSRC로 seq/ts 오프셋 연산을 통해 합칩니다. 수신 측 디코더는 "화자가 바뀌었다"는 사실을 전혀 모릅니다.

오프셋 기반 상대 연산

// 핵심 수식 (ptt_rewriter.rs)
virtual_seq = original_seq - origin_base_seq + virtual_base_seq
virtual_ts = original_ts - origin_base_ts + virtual_base_ts

화자 A가 seq=1000부터 시작하고, 화자 B가 seq=5000부터 시작해도, 가상 스트림에서는 연속된 seq로 변환됩니다. 패킷이 UDP 비순차 도착해도 원본의 상대적 순서가 보존됩니다.

Audio Rewriter
▸ ts_guard_gap = 960 (Opus 48kHz, 20ms)
▸ 키프레임 대기 없음 — 즉시 전환
▸ clear_speaker() 시 silence flush 3발
▸ dynamic ts_gap (경과 시간 × 48kHz)
Video Rewriter
▸ ts_guard_gap = 3000 (VP8 90kHz, ~33ms)
▸ 키프레임 대기 모드 — P-frame 드롭
▸ 새 화자의 첫 키프레임 도착 시 전환 완료
▸ PLI burst로 키프레임 요청

PTT 게이팅 — 비발화자 패킷 차단

Conference 모드에서는 모든 참가자의 패킷이 fan-out됩니다. PTT 모드에서는 현재 발화자의 패킷만 통과시키고, 나머지는 서버에서 gate 차단(drop)합니다.

이것이 텔레메트리에서 네트워크 loss vs gate loss 구분이 중요한 이유입니다. 클라이언트의 getStats()가 보고하는 "lost 157 packets"가 실제 네트워크 손실인지, 서버 gate 차단인지를 구분해야 정확한 진단이 가능합니다.

ingress_rtp_received = egress_rtp_relayed + ptt_rtp_gated + ptt_video_pending_drop + egress_drop

04 — ADAPTIVE TIMESTAMP ALIGNMENT

NetEQ Jitter Buffer를 제어하는 기술 — Dynamic ts_gap

PTT에서 가장 어려운 문제. 화자가 바뀔 때마다 수신 오디오 지연이 단조 증가하여 30초 만에 1초 이상 누적. 18시간 디버깅 세션, 3가지 시도 끝에 해결.

문제 — NetEQ Jitter Buffer 폭주

libwebrtc의 NetEQ는 arrival_time gap과 RTP timestamp gap의 불일치를 jitter로 해석합니다.

PTT에서 화자 A → 침묵 3초 → 화자 B 전환이 일어나면:

패킷은 3초 만에 도착하지만, 고정 ts_gap=960(20ms)이 적용되면 NetEQ는 "20ms 후 도착할 것으로 예상했는데 3000ms 후에 왔다 = jitter 2980ms"로 판단합니다.

→ buffer를 3초로 확장. 그리고 shrink=0 — 한 번 키운 buffer는 절대 줄이지 않음.

고정 ts_gap = 960 (20ms)
expected = 960/48000 = 20ms
actual (idle 3초) = 3000ms
jitter = 3000 - 20 = 2980ms
→ buffer 3초로 확장!
→ shrink=0: 절대 안 줄어듦
→ 전환 반복 시 무한 누적

해결 — Dynamic ts_guard_gap

핵심 아이디어: "ts gap을 실제 경과 시간에 맞추면?"

idle 구간이 3초였으면, ts gap도 3000ms × 48kHz = 144,000으로 설정합니다. 그러면 NetEQ가 계산하는 jitter는:

expected = 144000/48000 = 3000ms
actual = 3000ms
jitter = 0ms → buffer 안 키움!

이것은 RFC 3550 위반이 아닙니다. RTP timestamp는 샘플 카운터이지 벽시계가 아닙니다. "오래 침묵했다"를 의미하는 큰 ts gap은 완전히 합법적입니다.

dynamic ts_gap (경과시간 × 48kHz)
ts_gap = 3000ms × 48 = 144,000
expected = 144000/48000 = 3000ms
actual = 3000ms
jitter = 3000 - 3000 = 0ms
→ buffer 안 키움!
→ 매 전환마다 깨끗한 시작
실측 서버 로그
[PTT:REWRITE] idle=9147ms → ts_gap=439056
[PTT:REWRITE] idle=2575ms → ts_gap=123600
[PTT:REWRITE] idle=31881ms → ts_gap=1530288

실측 비교 — jbDelay 최대값

ITU-T G.114 기준: 150ms 이하 = 인지 불가, 150~400ms = 허용, 400ms 이상 = 대화 지장. Dynamic ts_gap의 97ms는 정상 범위 내.

보조 메커니즘 — Silence Flush 3발

화자 해제(clear_speaker) 시 Opus DTX silence 프레임 3발(60ms)을 fan-out합니다. 이것은 두 가지 역할을 합니다:

① NACK list full 방지

silence가 last_virtual_ts를 올려놓아 후속 dynamic ts_gap 계산의 기준점 역할.

② NetEQ에게 "스트림 끝" 신호

silence 프레임이 도착하면 NetEQ가 idle 모드로 전환할 준비를 함.

05 — VIDEO IN PTT MODE

PTT에서 비디오가 동작하는 방식

PTT는 음성만? 아닙니다. 0xLENS의 PTT는 발화자의 비디오도 함께 전환합니다. 오디오와 동일한 가상 SSRC 리라이팅이 비디오에도 적용되지만, 키프레임 대기라는 추가 메커니즘이 있습니다.

키프레임 대기 (Pending Keyframe)

비디오 디코더는 키프레임(I-frame)이 없으면 P-frame을 디코딩할 수 없습니다. 화자가 바뀌면 새 화자의 첫 키프레임이 도착할 때까지 이전 화자의 마지막 프레임이 표시됩니다.

① switch_speaker()

video rewriter: pending_keyframe = true. 이후 P-frame은 모두 drop.

② PLI burst

서버가 새 화자에게 PLI(Picture Loss Indication)를 발사하여 키프레임 생성을 요청.

③ 키프레임 도착

pending_keyframe = false. 이 시점부터 가상 SSRC로 리라이팅된 비디오 패킷이 fan-out.

CAMERA_READY — 클라이언트 주도 준비 프로토콜

모바일에서 카메라 콜드 스타트(AE/AWB 수렴)에 300~500ms가 걸립니다. 서버가 임의 타이밍에 PLI를 보내면 수렴 전의 어두운 프레임이 키프레임이 될 수 있습니다.

0xLENS는 클라이언트가 "카메라 준비 완료"를 선언(CAMERA_READY, op=18)한 후에 서버가 PLI를 발사하는 주도적 프로토콜을 사용합니다. 서버의 타이밍 추측이 불필요합니다.

PTT 전체 시퀀스

// 화자 A → 화자 B 전환 전체 시퀀스

SFU FloorController: IDLE → TAKEN(speaker=A)
SFU → 전체 WS FLOOR_TAKEN(op=141, speaker=A)
SFU ptt_rewriter.switch_speaker(A) — 오프셋 갱신
A mic unmute → RTP 전송 시작
SFU A의 RTP → rewrite(virtual SSRC) → fan-out
... (발화 중, Floor Ping 2초 주기) ...
A TALK 버튼 뗌 → WS FLOOR_RELEASE(op=41)
SFU FloorController: TAKEN → IDLE
SFU ptt_rewriter.clear_speaker() → silence flush 3발 + cleared_at 기록
SFU → 전체 WS FLOOR_IDLE(op=142)
... (침묵 구간, 3초 경과) ...
B TALK 버튼 누름 → WS FLOOR_REQUEST(op=40)
SFU FloorController: IDLE → TAKEN(speaker=B)
SFU ptt_rewriter.switch_speaker(B)

SFU video: pending_keyframe=true, PLI burst → B

B keyframe 도착 → pending=false → video fan-out 시작
