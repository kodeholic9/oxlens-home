Rust + Kotlin + JavaScript · Single UDP Port · ICE-Lite · No TURN

하나의 포트,
하나의 프로토콜,
세 개의 플랫폼.

0xLENS는 UDP 19740 단일 포트에서 STUN, DTLS, SRTP를 모두 처리하고, WS 1974 단일 포트로 시그널링을 수행합니다. TURN 서버 없이. STUN 외부 서버 없이. SDP 재협상 없이. 서버가 해야 할 일만 하고, 하지 말아야 할 일은 하지 않습니다.

02 — ICE-LITE (BUILT-IN STUN)

TURN이 필요 없는 이유, STUN 서버가 필요 없는 이유

일반적인 WebRTC 서비스는 STUN 서버(NAT 뒤의 공인 IP 확인용)와 TURN 서버(방화벽 우회 릴레이)를 별도로 운영합니다. 0xLENS는 둘 다 필요 없습니다. SFU 자체가 그 역할을 합니다.

TYPICAL WebRTC

별도 인프라 3개 필요

①STUN 서버 — 클라이언트가 자기 공인 IP를 알아내는 용도. NAT 뒤에 있으면 필수.
②TURN 서버 — 양쪽 다 NAT/방화벽 뒤일 때 미디어를 릴레이. 대역폭 비용 높음.
③SFU 서버 — 미디어 라우팅. 위 두 서버와 별도 운영.
운영 비용: STUN 서버 + TURN 서버 + SFU 서버 = 인프라 3중 관리
0xLENS ICE-LITE

SFU가 전부 처리

①Built-in STUN 응답 — SFU가 ICE-Lite로서 STUN Binding Response를 직접 생성. 클라이언트는 SFU를 ICE 상대로 인식.
②TURN 불필요 — SFU가 공인 IP에 위치. 클라이언트→SFU는 항상 직접 연결 가능 (outbound UDP).
③단일 바이너리 — STUN + DTLS + SRTP + 시그널링 + 릴레이 = 하나의 프로세스.
운영 비용: SFU 1대 = 끝. RPi에서도 동작.

ICE-Lite의 Built-in STUN — 어떻게 동작하는가

일반적인 WebRTC에서 클라이언트는 외부 STUN 서버(예: Google stun.l.google.com)에 Binding Request를 보내 자신의 NAT 매핑된 공인 IP:Port를 알아냅니다. 이것이 ICE candidate 수집 과정입니다.

0xLENS는 이 단계를 건너뜁니다. 클라이언트의 SDP에 iceServers: [] — 빈 배열을 넣습니다. 외부 STUN 서버를 안 쓴다는 뜻입니다.

대신 클라이언트는 SFU의 IP:Port를 직접 remote candidate로 받습니다. server_config 응답에 SFU의 공인 IP와 UDP 포트가 포함되어 있고, 클라이언트가 이것을 addIceCandidate()로 등록합니다.

그러면 브라우저의 ICE agent가 SFU로 직접 STUN Binding Request를 보냅니다. SFU의 ICE-Lite 구현(ice.rs)이 이 요청을 받아서:

① USERNAME 검증 (server_ufrag:client_ufrag)
② MESSAGE-INTEGRITY 검증 (HMAC-SHA1)
③ XOR-MAPPED-ADDRESS 생성 (클라이언트 공인 IP)
④ Binding Success Response 전송
⑤ USE-CANDIDATE → candidate pair selected

클라이언트 입장에서는 SFU가 ICE 상대이자 미디어 수신자입니다. RFC 8489 표준 프로토콜로 응답하므로 브라우저는 별도 STUN 서버와 차이를 구분하지 못합니다.

TURN이 불필요한 이유

TURN 서버가 필요한 경우는 양쪽 모두 NAT/방화벽 뒤에 있어서 직접 UDP 통신이 불가능할 때입니다. P2P 통화에서는 흔한 상황입니다.

하지만 SFU 구조에서는 상황이 다릅니다:

SFU = 공인 IP

SFU 서버는 공인 IP에서 UDP 포트를 열고 대기합니다. 클라이언트가 보내는 패킷은 outbound UDP이므로 대부분의 NAT/방화벽을 통과합니다.

NAT 매핑 유지

클라이언트가 SFU에 패킷을 보내면 NAT 매핑이 생기고, SFU의 응답(STUN Response, DTLS, SRTP)은 같은 매핑을 통해 돌아옵니다. ICE-Lite의 STUN 응답이 이 매핑을 최초에 만듭니다.

결과적으로, 클라이언트 → SFU(공인 IP) 방향의 outbound UDP만 성공하면 양방향 통신이 성립합니다. 기업 네트워크에서 outbound UDP가 완전 차단된 극단적인 경우만 예외이며, 이 경우에도 TURN보다는 TCP/TLS fallback이 더 현실적입니다.

"STUN 서버를 별도로 운영하는 건, 우체국에 가서 자기 집 주소를 물어보는 것과 같다. 우리 SFU는 편지를 받는 사람이 직접 주소를 알려준다."

03 — SINGLE PORT DEMUX

단일 UDP 포트에서 모든 것을 처리하는 법

UDP 19740 하나로 STUN, DTLS, SRTP를 모두 받습니다. 첫 바이트 1개만 보면 무엇인지 알 수 있습니다 (RFC 5764).

// demux.rs — RFC 5764 Section 5.1.2
First Byte
Type
Handler
0x00..0x03
STUN
ice.rs → Binding Response (ICE connectivity check)
0x14..0x3F
DTLS
dtls.rs → Handshake → SRTP 키 도출
0x80..0xBF
SRTP / SRTCP
srtp.rs → decrypt → fan-out → encrypt → send

연결 수립 시퀀스

1
WS 연결 + HELLO → IDENTIFY
클라이언트가 WS 1974에 연결. 서버가 HELLO(heartbeat_interval) 전송, 클라이언트가 IDENTIFY(user_id) 응답.
2
ROOM_JOIN → server_config
서버가 ICE credentials(ufrag/pwd), DTLS fingerprint, 코덱 정책, 공인 IP + UDP 포트를 전달. 이것이 STUN/TURN 서버를 대체하는 핵심.
3
Fake SDP 조립 → 2PC 생성
클라이언트가 server_config 기반으로 SDP를 로컬에서 직접 조립. SDP 교환이 아니라 SDP 빌딩. Publish PC, Subscribe PC 각각 생성.
4
ICE: STUN Binding (SFU가 응답)
브라우저 ICE agent가 SFU로 STUN Binding Request 전송. SFU의 ice.rs가 XOR-MAPPED-ADDRESS로 응답. 클라이언트의 NAT 매핑 성립.
5
DTLS Handshake → SRTP 키 도출
같은 UDP 포트에서 DTLS 핸드셰이크. 완료 후 RFC 5764 레이아웃으로 SRTP master key/salt 도출. 이 시점부터 미디어 암호화 가능.
6
SRTP 미디어 송수신
같은 UDP 포트로 SRTP 패킷 송수신 시작. 서버는 decrypt → 참가자 조회(by_addr, O(1)) → fan-out → encrypt → send.

04 — 2PC & SDP-FREE

SDP 재협상 지옥에서 탈출한 방법

일반적인 SFU는 참가자가 입장/퇴장할 때마다 SDP Offer/Answer를 재협상합니다. N명이 동시에 움직이면 SDP 재협상 경합(race condition)이 발생합니다. 이전 프로젝트를 말아먹은 원인이 바로 이것이었습니다.

SINGLE PC — 재협상 지옥

하나의 PeerConnection에 publish + subscribe 트랙을 모두 넣으면, 참가자 변동 시마다 전체 SDP를 다시 교환해야 합니다.

3명이 동시에 입장하면 SDP renegotiation 3건이 겹치고, 중간에 rollback이 필요하고, 타이밍에 따라 트랙이 누락되고...

→ "가끔 소리가 안 들려요" 버그의 근본 원인

2PC — 관심사 분리

Publish PC: 내 미디어를 서버에 보내는 전용. 한 번 만들면 다시 협상할 일 없음.

Subscribe PC: 남의 미디어를 서버에서 받는 전용. 참가자 변동 시 이것만 업데이트.

→ Publish는 절대 건드리지 않음. Subscribe만 관리.

SDP-Free 란?

서버가 SDP를 전혀 파싱하지 않습니다. 서버는 SSRC(32-bit 숫자)와 코덱 설정만 알면 됩니다.

SDP는 클라이언트 쪽에서 서버가 보내준 server_config를 기반으로 로컬에서 조립(Fake SDP)합니다. 서버→클라이언트 SDP 교환이 아니라, 서버가 재료를 주면 클라이언트가 레시피대로 만드는 구조입니다.

이 SDP 빌더는 JavaScript(oxlens-home), Kotlin(Android SDK) 두 곳에 동일한 로직으로 포팅되어 있습니다.

06 — DATA PLANE

미디어 처리 파이프라인

3단계 워커 파이프라인으로 30인 Conference loss 0.000%를 달성.

Ingress
▸ SO_REUSEPORT × 4 워커
▸ 커널이 4-tuple hash로 패킷 분배
▸ 각 워커: recv → demux → decrypt
▸ DashMap by_addr → O(1) 참가자 조회
Fan-out
▸ 같은 방 참가자에게 try_send()
▸ bounded mpsc (256 slots)
▸ 큐 풀 시 backpressure drop
▸ NACK/RTX가 손실 커버
Egress
▸ Subscriber별 전용 task
▸ 각 task가 outbound SRTP 독점
▸ Mutex 경합 제로 (LiveKit 패턴)
▸ encrypt → send_to()

O(1) 참가자 조회 — DashMap 3중 인덱스

미디어 hot path에서 가장 중요한 것은 "이 패킷이 누구의 것인가"를 빠르게 알아내는 것입니다.

by_addr : SocketAddr → (Participant, PcType) // SRTP hot path, 매 패킷
by_ufrag : String → (Participant, PcType) // STUN cold path, 연결 초기
by_user : String → Participant // 시그널링 (WS handler)

2PC 구조에서 같은 유저가 addr 2개를 사용(브라우저가 PC마다 별도 UDP 소켓 바인딩)하므로, by_addr과 by_ufrag에는 PcType(Pub/Sub) 구분이 포함됩니다.
